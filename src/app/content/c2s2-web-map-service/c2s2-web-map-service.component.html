Der Web Map Service bildet im Rahmen dieser Arbeit den ältesten OGC Standard, dessen Ursprung sich in der Version 1.0.0 vom 19.04.2000 findet. Die aktuelle Version 1.3.0 wurde am 15.03.2006 veröffentlicht und ist somit seit mehr als 15 Jahren unverändert.

Der OGC- WMS wurde zur Standardisierung von zahlreichen anderen Institutionen übernommen, hierzu gehören beispielsweise die International Organization for Standardization () oder auch die Infrastructure for Spatial Information in the European Community (). INSPIRE hat das Ziel, die Interoperabilität von Geodaten für alle Mitglieder der Europäischen Gemeinschaft zu standardisieren und mithilfe dessen den Datenaustausch für Nutzer zu vereinfachen. Insbesondere für diese Vorhaben ist es sinnvoll, sich auf bereits durch eine andere Organisation etablierte Standards zu stützen.

Sämtliche Karten werden üblicherweise in Layern gestaltet. Diese Methodik wird auch vom WMS unterstützt und ermöglicht es folglich, einzelne Layer an einen Client zu senden. Die folgende Abbildung veranschaulicht den Aufbau einer Karte:

Die Kommunikation zur Abfrage der Layer von einem Server, welcher den WMS bereitstellt, erfolgt über eine HTTP-Get-Methode. Die folgenden zwei Funktionen des Servers müssen dabei mindestens gegeben sein:

  • GetCapabilities = liefert alle Informationen als XML-Dokument, welche ein Client benötigt, um einen GetMap-Request zu tätigen
  • GetMap = dient der Auslieferung der Daten anhand der zuvor über GetCapabilities-Request ausgegebenen Daten

GetCapabilities bietet neben einer Auflistung aller vom Server zur Verfügung stehenden Layer auch weitere Angaben, wie beispielsweise Metainformationen über den Betreiber des Servers, verfügbare Koordinatensysteme, Formate, optionale Funktionen und Metadata der Datensätze.

Ein HTTP-GET-Request selbst stellt sich aus nachfolgend dargestellten Eingaben zusammen. Hierbei dient das Fragezeichen (?) der Aufteilung zwischen URL und übermittelten Parametern. Anhand dieser Parameter lässt sich erkennen, dass der WMS im Gegensatz zum TMS eine viel größere Möglichkeit bietet, Anpassungen vorzunehmen. Eine je nach Anwendungsfall wichtige Funktionalität ist es, durch den Server die Geodaten on-the-fly in ein anderes Koordinatensystem (CRS) projizieren zu lassen. Gleichzeitig lassen sich auch das Ausgabeformat (FORMAT) oder, über den Parameter STYLE, zusätzlich die Erscheinung des Datensatzes beeinflussen.

Die bisher beschriebenen Anfragen müssen innerhalb der Standardisierung des OGC vorgesehen sein. Neben diesen gibt es noch eine weitere optionale Anfrage, den GetFeatureInfo-Request. Dieser ermöglicht es, Informationen des Layers anhand der Pixelposition abfragen zu können. Der Request erfolgt mittels folgender Parameter:

Wichtig ist hierbei, dass die Parameter I (Zeile Spalte) und J (Zeilen Reihe) nicht die Koordinaten beinhalten, sondern die Position anhand der in WIDTH und HEIGHT angeforderten Pixel.

Durch den Parameter INFO_FORMAT kann die Rückgabe des Pixelwertes in einem für den jeweiligen Anwendungsfall passenden Format gewählt werden. Es empfiehlt sich, hierfür application/json zu verwenden, da dies eine Struktur beschreibt, welche von den gängigsten Programmiersprachen genutzt werden kann. Alternativ kann auch einfacher Text INFO_FORMAT=text/plain oder HTML INFO_FORMAT=text/html genutzt werden.

Alle hier exemplarisch genannten Aufrufe sind zwar funktional, wären jedoch für den täglichen Gebrauch eher ungeeignet, da die Requests von der Software selbst ausgeführt werden und der Nutzer sie folglich nicht selbst erzeugen muss.
